# NPFR Strategic Plan — `/splan`

Interactive web application for viewing, navigating, and tracking the North Port Fire Rescue Strategic Plan (FY2026–FY2029). Hosted on GitHub Pages at:

```
https://northportfirerescue.github.io/splan
```

---

## Repository structure

```
splan/

├── style.css                         # All styles — themes, pillar colors, layout
├── build.py                          # Pre-push build script (index + Excel)
├── README.md                         # This file
├── public/
│   └── Logo.png                      # NPFR shield logo (sidebar)

    └── NPFR_Strategic_Plan_YYYYMMDD.xlsx   # Generated by build.py
```

## Data model

All plan content lives in a single JSON file under `data/`. This file is the complete source of truth — it contains the full plan structure, all task statuses, and the changelog. Neither `index.html` nor `build.py` contain any hardcoded plan data.

```json
{
  "exported": "2026-06-09T00:00:00",
  "label": "Q2 review — updated P2 goals",
  "mission": "...",
  "pillars": [
    {
      "id": "P1",
      "name": "Operational Readiness",
      "steward": "Scott Lane",
      "purpose": "...",
      "goals": [
        {
          "id": "P1.26.G1",
          "year": "FY26",
          "predecessor": null,
          "owner": "TBD",
          "statement": "...",
          "metric": "...",
          "tasks": [
            { "text": "Gather incident data...", "status": "not-started" }
          ]
        }
      ]
    }
  ],
  "changelog": []
}
```

### Goal ID format

Goals use the format `P#.YY.G#` — pillar, two-digit fiscal year, goal number within that year. For example `P1.27.G2` is Pillar 1, FY27, second goal. This makes year membership visible in the ID itself and predecessor references self-documenting.

### Task statuses

Each task carries one of three statuses: `not-started`, `in-progress`, or `complete`. Goal-level status is derived automatically — all complete → Complete, all not-started → Not Started, anything else → In Progress.

## Application views

| View | Description |
|---|---|
| **Overview** | Six pillar cards with full descriptions, goal counts, gate counts, and per-fiscal-year task progress bars |
| **By Year** | All goals organized by fiscal year, grouped by pillar within each year with pillar color coding |
| **Pillars** | Full goal cards with tasks, status controls, dependency badges, and success metrics — grouped by pillar and fiscal year |
| **Dependencies** | Visual dependency chain map showing gate goals and what they unlock |

## Themes

Three themes switchable from the header:

| Theme | Use case |
|---|---|
| **Dark** | Default — matches the NPFR Community Risk Dashboard |
| **RED** | NPFR brand — crimson/maroon with gold accents from the shield logo |
| **Light** | Presentations and daylight viewing |

Theme preference is saved to localStorage.

## Edit mode

Click **✎ Edit** in the header to enter edit mode. This enables:

- **Pillar editing** — name, steward, purpose, ID
- **Goal editing** — statement, year, predecessor, pillar (move between pillars), owner, metric, ID
- **Task editing** — text, status, add/delete tasks
- **Adding goals and pillars** — add buttons appear in each year section and at the bottom of each pillar
- **Move a goal** — change the Pillar dropdown in the goal edit modal

Changes are tracked in a session changelog. Nothing persists until exported.

## Workflow

### Making and publishing status updates

```powershell
# 1. Start local server
cd C:\Users\rwilbert\splan
python -m http.server 8080
# Open http://localhost:8080

# 2. Update task statuses in the browser using the IP / ✓ buttons
#    Or use Edit mode to change goal/task text, move goals, etc.

# 3. Click ↑ Export in the header
#    Add a label describing what changed (e.g. "Q2 review — P5 goals updated")
#    Download the JSON file

# 4. Drop the JSON file into data/
#    The filename is statuses_YYYYMMDD.json — keep it dated

# 5. Run build.py to regenerate index.json and the Excel workbook
python build.py

# 6. Commit and push
git add .
git commit -m "status update YYYYMMDD — brief description"
git push

# GitHub Pages deploys automatically within ~30 seconds
```

### Structural changes (adding a pillar, renaming goals, etc.)

Use Edit mode in the browser, export the JSON, drop it in `data/`, run `build.py`, push.

### Historical snapshots

The snapshot dropdown in the header lists every `statuses_YYYYMMDD.json` file in `data/`. Select any date to view that snapshot as a read-only overlay. The plan does not go backwards — it is just a view. Click "Back to current" to return.

## build.py

Run before every `git push`. Reads the latest `statuses_*.json` from `data/` and:

1. Regenerates `data/index.json` listing all available snapshot files
2. Builds a themed Excel workbook saved to `exports/`
3. Prints a per-pillar completion summary to the terminal

```bash
python build.py
```

## Excel export

The **↓ Excel** button in the header serves the most recent pre-built workbook from `exports/`. It does not generate on the fly — run `build.py` first to ensure the file is current.

The workbook contains:
- **Cover** — agency name, mission, pillar summary with completion percentages
- **Summary** — all six pillars with task counts and completion rates
- **By Year** — all goals organized by fiscal year across all pillars
- **P1 through P6** — one sheet per pillar, goals grouped by fiscal year, all tasks with statuses
- **Dependencies** — all gate goal chains
- **Changelog** — full audit trail of status and structural changes

## Local development

No build step, no bundler, no Node.js required.

```bash
cd C:\Users\rwilbert\splan
python -m http.server 8080
# Open http://localhost:8080
```

Edit `index.html` or `style.css` directly and refresh. The app fetches `data/index.json` and the latest snapshot on load — this requires a server (`fetch()` does not work on `file://` protocol).
